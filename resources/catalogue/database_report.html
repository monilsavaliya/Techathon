Database Architecture Bible

Version 3.5.0 • Last Updated: Dec 17, 2025 • Classification: Confidential

The AI Sales Agent System is powered by a high-performance, file-based relational database architecture (FRDA). Unlike monolithic storage, this system distributes intelligence across 6 specialized JSON repositories, minimizing latency and ensuring Data Sovereignty for each autonomous agent.

This document serves as the single source of truth for the schema definitions, validation rules, and inter-agent data contracts.

1. Product Master Enriched (`product_master_enriched.json`)

The cornerstone of the Technical Agent. Contains over 4,000 SKUs with deep technical specifications. Used for O(1) Exact Matching and O(log n) Nearest Neighbor Search.

Schema Definition

Field Type Required Description
product_id PrimaryKey Yes Unique Identifier. Format: PC-{GRID}-{MAT}-{CORE}-{SIZE}
technical_specs Object Yes Container for all matching parameters (Voltage, Insulation, etc).
commercial.base_mfg_cost Float Yes The raw ex-factory cost. CRITICAL: Not visible to Sales Agent.
inventory_stock Integer Yes Current physical stock. Triggers "Check Factory" logic if 0.
EXAMPLE 1: High Voltage Aluminium Cable (Infrastructure)
{
  "product_id": "PC-HT-AL-3C-400",
  "technical_specs": {
    "voltage_grade": "11kV",
    "conductor_material": "Aluminium",
    "insulation": "XLPE",
    "armour_type": "Flat Strip",
    "standards": ["IS 7098 Part-2"]
  },
  "commercial": {
    "base_mfg_cost_per_meter": 4500.0,
    "inventory_stock": 50
  }
}
EXAMPLE 2: Low Voltage Copper Flexible (Domestic/Industrial)
{
  "product_id": "PC-LT-CU-1C-2.5",
  "technical_specs": {
    "voltage_grade": "1100V",
    "conductor_material": "Copper",
    "core_count": 1,
    "cross_section_sqmm": 2.5
  },
  "commercial": {
    "base_mfg_cost_per_meter": 45.0,
    "inventory_stock": 10000 // High volume SKU
  }
}

2. Competitor Intelligence (`competitors.json`)

Provides the bounding constraints for "Game Theory Pricing". The system assumes competitors act rationally to maximize their own Win Rate.

Strategic Fields

Field Type Logic
competitor_id PK Internal reference ID (e.g., COMP-001).
colliding_skus ForeignKey[] List of product_ids where they directly compete.
pricing_intelligence.aggression_score 1-10 Score 1-4: Ignore.
Score 5-7: Match Price.
Score 8-10: Undercut by 2%.

Usage in Code (pricing_agent.py)

# Game Theory: Nash Equilibrium Adjustment
if competitor['pricing_intelligence']['aggression_score'] > 7:
    # Competitor is Aggressive -> We sacrifice margin to win
    target_margin = 0.04  # Drop to 4% "Survival Cost"
    strategy_note = "Defensive Pricing Triggered"
else:
    target_margin = 0.15  # Maintain 15% Healthy Margin

3. Client Master (`client_master.json`)

Dictates the "Risk Premium" added to the base margin.

Business Logic

A "Tier-1" client gets a 5% discount but stringent penalty clauses. A "Tier-3" client pays full price but gets flexible payment terms.

EXAMPLE 1: Tier-1 Infrastructure Major (Strategic)
{
  "client_id": "CL-ORN-001",
  "tier": "Tier-1 (Infrastructure Major)",
  "loyalty_status": "Gold",
  "payment_terms": "90 Days Credit",
  "risk_profile": "Low"
}
EXAMPLE 2: Tier-3 Contractor (Transactional)
{
  "client_id": "CL-SML-992",
  "tier": "Tier-3 (Private Contractor)",
  "loyalty_status": "New",
  "payment_terms": "100% Advance",
  "risk_profile": "High"
}

Usage in Code (pricing_agent.py)

# Determine Risk Premium based on Client Tier
if client['payment_terms'] == "90 Days Credit":
    interest_cost = total_price * 0.03  # 3% Cost of Capital
    final_price += interest_cost
elif client['tier'] == "Tier-3":
    margin = margin + 0.05 # Add 5% Risk Buffer

4. Factory Production Schedule (`factory_production_schedule.json`)

Mimics a read-only feed from the Manufacturing Execution System (MES).

Validation Rules

  • Line Utilization > 100%: Impossible. System throws validation error.
  • Utilization > 90%: Triggers "Surge Pricing" (+5% Markup) and "Lead Time Extension" (+14 Days).

5. Logistics Master (`logistic_master.json`)

Geospatial pricing logic. Defines costs based on Zonal Complexity.

6. Central RFP Database (`central_rfp_database.json`)

The State Object. This file is transactional and high-velocity. It uses a "Stage Tracker" to ensure ACID-like consistency across agents.

State Machine Transition

Approved

Usage in Code (main_agent.py)

# The "Baton Pass" State Machine
if rfp['processing_stage']['tech_agent'] == "Completed":
    # Only run Pricing if Tech is done
    run_pricing_agent(rfp_id)
    update_state(rfp_id, "Pricing_In_Progress")
else:
    # Block execution to prevent data corruption
    raise StateError("Dependency checklist failed")
